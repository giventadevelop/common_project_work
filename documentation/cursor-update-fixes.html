🖱️ Cursor IDE — Update & Install Fixes

Canonical playbook for fixing Cursor update/install failures on the HP Omen (gain). Built from repeated real fixes.
The one-line summary: Cursor's in-app updater fails because it tries to replace files (Cursor.exe, node.exe) while Cursor or its helper processes are still running. The reliable fix is always: kill every Cursor-related process, then run the standalone installer with Cursor closed.

📌 Standard Operating Procedure (do this every update)

  1. Don't use the in-app updater (Help → Check for Updates). It forces full restarts and repeatedly fails. Use the manual installer instead.
  2. Download the latest user-setup x64 installer from Cursor's official server (see "Getting the installer" below). Save to E:\Categories\Software_Installs.
  3. Kill ALL Cursor-related processes (see the kill script). This includes the sneaky node.exe helpers inside the Cursor folder.
  4. Do NOT reopen Cursor until the install fully completes — launching it mid-install respawns the helpers and re-locks files.
  5. Run the installer: right-click → Properties → Unblock if shown → run it (user-setup needs no admin).
  6. Verify the installed version afterward (Help → About).

🚨 Known errors & their fixes

"The process cannot access the file because it is being used by another process" file lock

Installer can't replace Cursor.exe. Cause: Cursor (or a leftover process) is still running.

Fix: Kill all Cursor processes (main app + helpers + inno_updater.exe), then click Retry.

"Failed to create file handle" / "verify there are no Cursor processes still executing" file lock

Same root cause. There are often 30+ Cursor.exe helper/renderer/GPU/extension-host processes alive even when the window is closed. Plus inno_updater.exe (the Inno Setup updater helper) can hold the handle.

Fix: Kill everything matching cursor or inno_updater, confirm 0 remain, then Retry.

"delete file failed… access is denied: node.exe" node lock

(The filename may appear garbled in the dialog, e.g. "Nod.Dexe" — that's node.exe.) Cause: Cursor's bundled Node helper processes are still running from inside the install folder: ...\cursor\resources\app\resources\helpers\node.exe. They survive after the Cursor window closes, and because they're named node (not cursor), a normal "close Cursor" and even a cursor-only process kill miss them.

Fix: Kill only the node.exe processes whose path is under the cursor folder — never blanket-kill all node.exe (that would take down dev servers, Adobe, etc.). Then Retry.

"The latest version of Cursor is already installed" but it's broken / older version skip

The reinstaller compares versions and skips if yours is equal-or-newer, doing nothing. Happens after a half-finished update leaves mismatched files.

Fix: Download a genuinely newer build, or do a full uninstall + clean reinstall.

🔧 The kill script (targeted & safe)

Kills Cursor, its in-folder node helpers, and inno_updater — while leaving your other node.exe (dev servers, Adobe Creative Cloud, global nodejs) untouched. Run in PowerShell.

# Kill Cursor app + in-folder node helpers + inno_updater (targeted)
$cursorNode = Get-Process node -ErrorAction SilentlyContinue |
    Where-Object { $_.Path -like "*\Local\Programs\cursor\*" }
$cursorApp  = Get-Process -ErrorAction SilentlyContinue |
    Where-Object { $_.ProcessName -match 'cursor|inno_updater' }

($cursorNode + $cursorApp) | Stop-Process -Force -ErrorAction SilentlyContinue
Start-Sleep -Seconds 2

# Confirm clean (want 0 and 0)
$nodeLeft = (Get-Process node -ErrorAction SilentlyContinue |
    Where-Object { $_.Path -like "*\Local\Programs\cursor\*" } | Measure-Object).Count
$appLeft  = (Get-Process -ErrorAction SilentlyContinue |
    Where-Object { $_.ProcessName -match 'cursor|inno_updater' } | Measure-Object).Count
"CURSOR_NODE_LEFT=$nodeLeft  CURSOR_PROCS_LEFT=$appLeft"

✅ Pre-retry verification (optional but smart)

Confirms the lock is actually released and the folder is writable before you hit Retry.

$node = "$env:LOCALAPPDATA\Programs\cursor\resources\app\resources\helpers\node.exe"
try { $fs=[System.IO.File]::Open($node,'Open','ReadWrite','None'); $fs.Close(); "NODE=UNLOCKED" }
catch { "NODE=LOCKED" }
$dir = "$env:LOCALAPPDATA\Programs\cursor"
try { "x"|Out-File "$dir\_t.tmp"; Remove-Item "$dir\_t.tmp" -Force; "FOLDER=WRITABLE" }
catch { "FOLDER=NOT_WRITABLE" }

⬇️ Getting the installer (official, verifiable)

Cursor's installers live on downloads.cursor.com under a per-release commit hash. This machine is user-setup, x64 — match that.

Build typeURL pattern
User (this machine).../production/<commit>/win32/x64/user-setup/CursorUserSetup-x64-<ver>.exe
System-wide.../production/<commit>/win32/x64/system-setup/CursorSetup-x64-<ver>.exe
  • Find the latest version + commit hash at cursorhistory.com or oslook/cursor-ai-downloads version-history.json. The site usually shows the system link — swap system-setup/CursorSetup → user-setup/CursorUserSetup for the user build.
  • A genuine build is ~186 MB. A tiny file = an error page, not the installer.
  • Avoid winget for Cursor: the Anysphere.Cursor package is frequently stale and known to report success while installing nothing.

📝 Download & checksum snippet

$ver = '3.7.36'
$commit = '776d1f9d76df50a4e0aeca61819a88e7c1b861e2'   # update per release
$url = "https://downloads.cursor.com/production/$commit/win32/x64/user-setup/CursorUserSetup-x64-$ver.exe"
$out = "E:\Categories\Software_Installs\CursorUserSetup-x64-$ver.exe"
Invoke-WebRequest -Uri $url -OutFile $out
"SIZE=" + (Get-Item $out).Length
"SHA256=" + (Get-FileHash -Algorithm SHA256 $out).Hash

Note: chocolatey often lags the newest version, so a second-source checksum may not exist for brand-new builds. Provenance (official HTTPS host + release commit hash + ~186 MB size) is the practical verification in that case.

🔁 Post-install

  • Confirm version: Cursor → Help → About.
  • Settings, extensions, and .cursor\mcp.json are preserved by the user-setup installer.
  • If an MCP server (e.g. Cipher-MCP) shows disconnected on first launch, restart Cursor once — normal MCP handshake-timeout, reconnects on the second start.
