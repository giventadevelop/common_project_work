👥 GitHub Organization Configuration Guide

Complete playbook for giventadeveloporg — seats, members, teams, repository access, and protected release/* branch merges. No credentials or secrets stored here.
The one-line summary: GitHub access is layered. A user needs (1) an org seat and membership, (2) Write or Maintain on each repository via a team, and (3) a place on the ruleset bypass list to merge into protected release/* branches. Team membership alone does not grant write access across all org repositories.

🧩 The three permission layers

Layer 1 — Org membership + seat
  User is invited and accepts → counts against licensed seats

Layer 2 — Repository access (Write / Maintain)
  Team is linked to each repo with Write or Maintain → user can push branches and merge PRs

Layer 3 — Ruleset bypass (release/*)
  giventa_team on bypass list of release_branch_ruleset → user can work on protected release branches

All three layers are required for a developer to open a PR into release/v1.0.0 (or any release/* branch) and merge it.

📋 Standard onboarding checklist (new developer)

  1. Add a licensed seat (if none available)
  2. Invite user to the organization
  3. User accepts the invitation
  4. Add user to giventa_team
  5. Confirm giventa_team has Write or Maintain on every repo they need
  6. Confirm giventa_team is on the Bypass list of release_branch_ruleset
  7. Verify effective permission with gh api (see Verification section)

Step 1 — Add a licensed seat

Required when inviting returns HTTP 422: “No seats are available to invite this user.”

  1. Sign in as an organization owner.
  2. Open /organizations/giventadeveloporg/settings/licensing
  3. Review seats used vs seats purchased.
  4. Click Add seats and purchase one additional seat.
  5. Alternatively, remove an inactive member or cancel a pending invitation to free a seat.

Error: “No seats are available to invite this user” HTTP 422

Fix: Add a seat at the licensing URL above, or free an existing seat, then retry the invitation.

Step 2 — Invite a new organization member

GitHub CLI

gh api orgs/giventadeveloporg/invitations \
  -f email=NEW_USER@example.com \
  -f role=direct_member

GitHub web UI

  1. Organization → Settings → Members
  2. Invite member → enter email or GitHub username → role Member

Successful invitation (response shape) reference

{
  "email": "NEW_USER@example.com",
  "role": "direct_member",
  "failed_at": null,
  "team_count": 0
}

"team_count": 0 confirms the user was not added to any team automatically. Continue to Step 3.

Invite and assign team in one step (optional)

# Get team ID
gh api orgs/giventadeveloporg/teams -q '.[] | select(.slug=="giventa_team") | .id'

# Invite with team (replace TEAM_ID)
gh api orgs/giventadeveloporg/invitations \
  -f email=NEW_USER@example.com \
  -f role=direct_member \
  -F team_ids[]=TEAM_ID

Step 3 — Add user to the developer team

Org invitation does not add the user to giventa_team unless you explicitly do so (or used team_ids during invite).

Web UI

  1. Open giventa_team
  2. Members tab → Add member
  3. Search for their GitHub username (they must have accepted the org invite first)

GitHub CLI

gh api -X PUT orgs/giventadeveloporg/teams/giventa_team/memberships/GITHUB_USERNAME \
  -f role=member

Step 4 — Release branch ruleset bypass

Organization ruleset release_branch_ruleset protects all release/* branches across all repositories.

Where to manage bypass

  1. Organization → Settings → Rules → Rulesets
  2. Open release_branch_ruleset
  3. In Bypass list, click + Add bypass
  4. Add a Team (preferred: giventa_team) or individual User
  5. Set permission to Always allow

Step 5 — Grant repository Write access via team

Critical fix: merge denied on one repo but works on another mosc-temp case

A user was on giventa_team and on the ruleset bypass list, but could merge to release/* in strapi-editorial-template and not in mosc-temp.

Root cause: Repository permissions differed per repo. Org default permission is Read. giventa_team was not linked to every repository with Write/Maintain.
RepositoryUser effective permission (before fix)Merge to release/*
strapi-editorial-templateWrite / MaintainAllowed
mosc-tempRead onlyDenied
Fix: Add giventa_team to mosc-temp (and all other dev repos) with Write or Maintain.

Web UI — assign repos to the team

  1. Open giventa_team → Repositories
  2. Add repository
  3. Add each repo the team should access:
    • mosc-temp
    • strapi-editorial-template
    • malayalees-us-site-boot
    • event-site-manager-batch-jobs
  4. Set role to Write or Maintain for each

GitHub CLI — add one repo to the team

# Write access (permission=push)
gh api -X PUT orgs/giventadeveloporg/teams/giventa_team/repos/giventadeveloporg/mosc-temp \
  -f permission=push

# Maintain access
gh api -X PUT orgs/giventadeveloporg/teams/giventa_team/repos/giventadeveloporg/mosc-temp \
  -f permission=maintain

Alternative — per-repo access

  1. Repo → Settings → Collaborators and teams
  2. Add teams → select giventa_team → Write or Maintain

✅ Verification commands

Replace GITHUB_USERNAME with the developer’s GitHub login. Run after onboarding or when debugging access.

gh api orgs/giventadeveloporg/members --paginate -q ".[].login"

gh api orgs/giventadeveloporg/teams/giventa_team/repos --paginate \

gh api repos/giventadeveloporg/mosc-temp/collaborators/GITHUB_USERNAME/permission
gh api repos/giventadeveloporg/strapi-editorial-template/collaborators/GITHUB_USERNAME/permission

# Expected for merge access: permission is "write" or "maintain" (not "read")

# Inspect release ruleset bypass actors
gh api repos/giventadeveloporg/mosc-temp/rulesets/13472925 \
  -q ".bypass_actors"

🚨 Troubleshooting

Cannot invite user — no seats available 422

Fix: Add a seat or remove unused member / pending invite.

Merge to release/* denied — “don’t have permission” repo access

Check: gh api repos/giventadeveloporg/REPO/collaborators/USERNAME/permission
Fix: Add giventa_team to the repo with Write or Maintain (Step 5).

Can push feature branch but cannot merge PR to release ruleset

Check: User or their team is on the Bypass list of release_branch_ruleset.
Fix: Add giventa_team to bypass with Always allow (Step 4).

Works in one repo, fails in another inconsistent team repos

Cause: Team is not linked to all repos, or repos were granted access individually and inconsistently.
Fix: Add all org dev repos to giventa_team with the same Write/Maintain role.

User not found when adding to team pending invite

Cause: User has not accepted the organization invitation yet.
Fix: Wait for acceptance, then add to team.

🔒 Security notes

  • Do not store passwords, personal access tokens, or API keys in documentation or git.
  • Grant Owner org role only to trusted administrators.
  • Prefer team-based access over per-user repo assignments for consistency.
  • Review licensed seats periodically at the licensing settings page.
  • Ruleset bypass grants elevated ability on release/* — keep the bypass list limited to trusted teams.
